# 09 - Git Part 3b: AI와 함께하는 브랜치

**이 섹션에서는 동일한 브랜치 작업을 다시 해봅니다 — 하지만 이번에는 명령어를 직접 입력하는 대신 AI와 대화합니다.** 브랜치, 전환, 병합이 무엇인지 이해했으니, AI가 문법을 처리하도록 하고 여러분은 하고 싶은 것에 집중하면 됩니다.

---

## 1. 소개

이전 섹션에서는 브랜치 관련 명령어를 모두 직접 타이핑했습니다. 이제 브랜치가 평행 타임라인이라는 것, 타임라인 사이를 전환하는 방법, 다시 합치는 방법을 이해했습니다.

로컬 git과 리모트와 마찬가지로, 이 명령어들을 외울 필요가 없습니다. "병합"이 무엇인지 알면 AI에게 "experiment 브랜치를 main에 합쳐줘"라고 말할 수 있고, AI가 적절한 명령어를 알아냅니다.

---

## 2. 브랜치 작업을 위한 프롬프트 예시

자연어가 배운 브랜치 명령어와 어떻게 대응되는지 살펴봅시다:

| 하고 싶은 것 | AI에게 말하는 내용 | 실행되는 명령어 |
|---|---|---|
| 새 브랜치 만들기 | "experiment라는 브랜치를 만들어줘" | `git branch experiment` + `git switch experiment` |
| 브랜치 전환 | "main 브랜치로 전환해줘" | `git switch main` |
| 모든 브랜치 보기 | "모든 브랜치를 보여줘" | `git branch` |
| 브랜치 병합 | "experiment를 main에 병합해줘" | `git switch main` + `git merge experiment` |
| 브랜치 삭제 | "experiment 브랜치를 삭제해줘" | `git branch -d experiment` |
| 만들면서 전환 | "feature라는 새 브랜치를 만들고 전환해줘" | `git switch -c feature` |
| 현재 브랜치 확인 | "지금 어떤 브랜치에 있어?" | `git branch` |

> **참고:** 정확히 이 문구를 사용할 필요는 없습니다. "새 브랜치 만들어줘", "메인 브랜치로 가줘", "experiment를 main에 합쳐줘" 등 다 통합니다.

---

## 3. 가이드 실습

LLM CLI 도구를 사용하여 Part 3a의 시나리오를 다시 해봅시다.

### 1단계: LLM CLI 도구 시작하기

iTerm2를 열고 `my-todo-list` 폴더로 이동한 다음 도구를 실행합니다:

```bash
cd my-todo-list
```

설치한 도구를 시작합니다:

| 도구 | 명령어 |
|------|--------|
| Claude Code | `claude` |
| Codex | `codex` |
| Gemini CLI | `gemini` |

> **참고:** [Claude Code](https://docs.anthropic.com/en/docs/claude-code/overview) | [OpenAI Codex CLI](https://github.com/openai/codex) | [Gemini CLI](https://github.com/google-gemini/gemini-cli)

### 2단계: 브랜치 확인

다음과 같이 입력합니다:

> "모든 브랜치를 보여주고 지금 어떤 브랜치에 있는지 알려줘"

AI가 `git branch`를 실행하고 결과를 보여줍니다.

### 3단계: 새 브랜치를 만들고 전환하기

> "experiment라는 새 브랜치를 만들고 전환해줘"

AI가 `git switch -c experiment` (또는 동등한 명령어)를 실행합니다.

### 4단계: 변경하고 커밋하기

VSCode에서 `TODO.md`를 편집합니다 — "Stretch Goals" 섹션을 추가합니다 (또는 원하는 내용을 추가하세요). `Cmd+S`로 저장합니다. 그런 다음:

> "TODO.md의 변경 사항을 'Add stretch goals section'이라는 메시지로 커밋해줘"

AI가 `experiment` 브랜치에서 스테이징하고 커밋합니다.

### 5단계: main으로 돌아가기

> "main 브랜치로 돌아가줘"

AI가 `git switch main`을 실행합니다. VSCode에서 `TODO.md`를 열어보세요 — 변경 사항이 사라졌습니다 (다른 브랜치에 있기 때문입니다).

### 6단계: 브랜치 병합하기

> "experiment 브랜치를 main에 병합해줘"

AI가 `git merge experiment`를 실행합니다. `TODO.md`를 열어보세요 — 변경 사항이 돌아왔습니다!

### 7단계: 정리하기

> "experiment 브랜치를 삭제해줘"

AI가 `git branch -d experiment`를 실행합니다.

## 4. 핵심 통찰

Part 1b, 2b와 같은 패턴을 주목하세요: Part 3a와 정확히 같은 브랜치 작업 — branch, switch, merge, delete — 을 했지만, 명령어를 기억할 필요가 없었습니다.

**개념을 이해하는 것이 중요합니다:**

- **브랜치** = 프로젝트의 평행 타임라인
- **전환 (Switch)** = 타임라인 사이를 이동
- **병합 (Merge)** = 한 타임라인을 다른 타임라인에 합치기
- **삭제 (Delete)** = 더 이상 필요 없는 브랜치 정리

이 개념들을 이해하면, 원하는 것을 자연어로 표현할 수 있습니다. AI가 여러분의 의도를 올바른 git 명령어로 변환합니다.

**브랜치 작업에서 AI 도움이 특히 빛납니다.** 명령어가 까다로워질 수 있기 때문입니다 — 특히 병합 충돌, 리베이스, 여러 브랜치 관리 같은 상황에서요. 세부 사항은 AI에게 맡기고 여러분은 의도에 집중하세요: "내 작업을 합쳐줘" 또는 "새로운 실험을 시작해줘."

## 5. 워크숍 정리

축하합니다! git 워크숍의 세 파트를 모두 완료했습니다. 배운 모든 것을 정리합니다:

### Part 1: 로컬 Git

| 개념 | 의미 |
|------|------|
| **스냅샷 (커밋)** | 프로젝트의 특정 시점 저장본 |
| **스테이징** | 다음 스냅샷에 넣을 파일 고르기 |
| **타임라인 (로그)** | 모든 스냅샷의 이력 |
| **Diff** | 정확히 무엇이 바뀌었는지 보기 |
| **복원** | 커밋하지 않은 변경 사항 되돌리기 |

### Part 2: 리모트

| 개념 | 의미 |
|------|------|
| **리모트** | 다른 컴퓨터(GitHub 같은)에 있는 프로젝트 사본 |
| **푸시** | 내 커밋을 리모트에 업로드 |
| **풀** | 리모트의 커밋을 내 컴퓨터에 다운로드 |

### Part 3: 브랜치

| 개념 | 의미 |
|------|------|
| **브랜치** | 실험을 위한 평행 타임라인 |
| **전환 (Switch)** | 타임라인 사이를 이동 |
| **병합 (Merge)** | 타임라인을 합치기 |

### 핵심 교훈

모든 git 명령어를 외울 필요가 없습니다. 중요한 것은 개념을 이해하는 것입니다 — 스냅샷, 타임라인, 리모트, 브랜치. 머릿속에 개념 모델이 있으면, 직접 명령어를 입력하거나 AI에게 대신 해달라고 요청할 수 있습니다.

**개념을 배우세요. 문법은 AI에게 맡기세요.**
